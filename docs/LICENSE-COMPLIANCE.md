# 许可证合规（一票否决项的核查结论）

核查日期：2026-07-17。上线前如引擎许可证有变动需复核。

## 结论：绿灯，但有三条架构铁律

| 引擎 | 许可证 | 托管 SaaS | 自部署分发 | 关键义务 |
|---|---|---|---|---|
| new-api (QuantumNous) | **AGPL-3.0** + 附加条款 | 允许（跑原版镜像） | 允许（引用官方镜像，不打包进我们的发行物） | 修改后提供网络服务必须开源修改；UI 须保留 "Frontend design and development by New API contributors." 署名与项目链接 |
| sub2api (Wei-Shaw) | **LGPL-3.0** | 允许 | 允许 | 分发修改版需公开对 sub2api 本体的修改（公开 fork 即可）。README 中"从未授权商业运营"为免责声明（撇清责任），非许可条款；法律上以 LICENSE 为准 |
| relay-panel 本体 | MIT | — | — | — |

## 三条架构铁律

1. **引擎零修改原则**。编排器只通过引擎的公开 admin API / 配置文件操作引擎，永远拉取官方发行版（docker image / release 二进制）。需要引擎级定制时：
   - sub2api → 维护**公开 fork**（LGPL 义务即满足）
   - new-api → 优先在编排层实现；确实绕不开则公开 fork（AGPL 义务）或联系 support@quantumnous.com 商谈商业授权
2. **不打包引擎进我们的发行物**。自部署版的 compose 文件通过 image 引用官方镜像，AGPL/LGPL 代码不进入我们的仓库和镜像，避免"聚合发行"的传染争议。
3. **编排器与引擎进程隔离**。编排器是独立程序，经网络 API 与引擎通信 —— 这是 AGPL/LGPL 语境下"独立作品"的最强姿态。禁止把引擎代码 import 进编排器进程。
